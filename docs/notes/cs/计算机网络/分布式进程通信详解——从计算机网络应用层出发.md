 分布式进程通信需要解决的问题有三：
 1. 进程标识和寻址问题（如何建立对等层通信连接）
 2. 传输层向应用层提供了怎样的服务
 3. 传输层-应用层的SAP是如何工作的（应用层如何利用传输层服务）

## 进程标识和寻址问题
进程的标识与寻址是借助于IP地址和端口号实现的。
IP地址用于标识主机，端口号则是用于标识主机上的特定应用进程。
对于一些知名的应用服务，有一些特定（预留）的端口号，例如Web服务器用端口号80进行标识，邮件服务器进程（使用SMTP协议）用端口号25标识。

## 传输层向应用层提供了怎样的服务
我们这里仅探讨*TCP/IP网络*提供的传输服务，那么传输层一共只向上层提供两种服务：==TCP提供的服务和UDP提供的服务==。在详细展开两种服务之前，我们先来了解应用程序需要的服务（或者也可以说成可供应用程序使用的运输服务）的分类。值得说明的是，这个分类仅仅是一种经验说法，并不是某种规范。
1. **可靠数据传输**：数据在传输过程中会不会*丢失、乱序、多出冗余噪声等*
2. **吞吐量**：能否以某种特定的速率交付数据报文。
3. **控制延迟**：能够保证在某个时间延迟内交付完整的数据报文。
4. **安全性**：数据机密性、数据完整性、端点鉴别等。

| 应用            | 数据丢失 | 带宽需求 | 时间敏感          |
| ------------- | ---- | ---- | ------------- |
| 文件传输/下载       | 不能丢失 | 弹性   | 不             |
| 电子邮件          | 不能丢失 | 弹性   | 不             |
| Web文档（网页）     | 不能丢失 | 小    | 不             |
| 因特网电话/视频会议    | 容忍丢失 | 较大   | 敏感，100ms以内    |
| 流媒体应用         | 容忍丢失 | 较大   | 敏感，几秒以内       |
| 交互式游戏         | 容忍丢失 | 小    | 敏感，10-100ms以内 |
| 智能手机讯息/社交媒体应用 | 不能丢失 | 弹性   | 灵活            |
*注：所有的应用都当然希望越安全越好*


### TCP服务
TCP服务模型是一种==面向连接的服务==，是属于==可靠数据传输的服务==。当某个应用程序调用TCP作为其运输层协议时，该应用程序就能获得来自TCP的可靠数据传输的服务。
- **面向连接的服务**，指的是在==数据传输之前==，TCP会让发送方和接收方相互交换**运输层控制信息**，即**握手**过程。这个握手过程是在提醒双方，==要为即将来临的大量分组传输做好准备==。在握手过程之后，一条**全双工的**TCP连接就建立了。当应用程序结束报文（message）发送后，这条连接将会被拆除。
- TCP还具有**拥塞控制机制**，这种服务不一定能为通信进程带来好处，但是能为整个互联网带来好处。当发送方和接收方之间的网络出现拥塞时，TCP的**拥塞控制机制**会抑制发送进程。同时，它也会限制每个TCP连接，使他们达到公平共享网络带宽的目的。

| 通信类型  |               特征                |
| :---: | :-----------------------------: |
| 全双工通信 |         双方均可以同时发送和接收数据          |
| 半双工通信 | 双方均可以发送和接收数据，但同一时间内只能有一方发送、一方接收 |
| 单工通信  |          只有一方能发送、另一方接收          |

### UDP服务
UDP服务是一种不提供不必要服务的轻量级运输协议，仅提供最低限度的服务（或者说尽力而为的服务）。与TCP相对地，UDP提供的运输服务是不可靠的。
- **无连接的服务**，指的是正式通信之前进程之间没有握手过程。因此数据可能会*丢失、冗余或乱序*。
- **不包含拥塞控制机制**，这意味着UDP的发送端可以用任意它选定的速率向下层注入数据。（注意的这个数据不代表接收端接收到的数据，即这个并不等同于吞吐量，原因自然是因为复杂的网络核心！）

## 应用层如何利用传输层服务
### 基本原理
分布式进程的通信过程特别像*邮寄快递*的过程，寄快递我们需要：
1. 将货物交给快递小哥
2. 标明寄件人的信息和住址
3. 标明收件人的信息和住址
而在分布式进程通信的过程中，同样也是考虑这么三个动作：
- 要传递的信息报文（message）/SDU是什么
- 发送方的主机与端口信息
- 接收方的主机与端口

### Socket API的引入
前文说过，只需要主机**IP地址**和应用进程对应的**端口号**就足以标识和寻址相互通信的分布进程。不过我们很自然会想到一个问题——如果每次需要通信都需要应用程序编写者（而不是计算机网络与操作系统）手动编写这些通信所需的信息，负担未免过于大。

于是操作系统承担了这一部分的抽象工作，将进程与计算机网络之间的接口单独抽象为操作系统提供的**应用编程接口**（API），这就是我们所谓的**Socket API**，中文叫做**套接字接口**。

Socket的工作原理类似于 `open()` 系统调用产生的==“句柄“==，操作Socket相当于操作TCP/UDP连接本身。直观上他就是一个整数，而本质上是==TCP/UDP连接产生的会话关系的本地标识==。对于TCP服务，Socket是一个四元组，标识发送方和接收方各自的主机IP地址与进程端口号。而对于UDP服务，Socket是一个二元组，标识发送方的主机IP和端口号。

当分布进程需要相互通信时，发送方只需要将==报文和对应的TCP套接字==发送到层间的服务访问点，从应用层的角度就已经完成任务了，UDP类似，只是还需要手动配置接收方的主机IP和端口号。除了编程上的便利外，引入套接字至少还有这么些好处：
1. 穿过层间的信息量减少
2. 易于操作系统管理各种连接

以上信息密度比较大，我们归纳几个要点：
- 套接字作为一个”本地“的标识，实际上指的是，这个套接字是由==操作系统==进行分配，而非会话本身的某种固定属性。因此同一个会话的双方操作系统分配的套接字可以是不同的值。
- TCP套接字是一个四元组，UDP套接字是一个二元组，它们的工作原理类似于操作系统文件操作的”句柄“。
- 在”协议层间数据传输模型“中，不同传输层协议（例如TCP和UDP）对应的套接字不同，因此也就对应不同的服务访问点，从而传到下层的不同服务提供者。
