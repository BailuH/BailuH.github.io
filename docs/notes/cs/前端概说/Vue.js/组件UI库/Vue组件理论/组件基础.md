## 概说
我们来看这么一个组件：
```vue
<script setup lang="ts">
import { type ArticleDocumentResponse, DefaultService } from '@/client'
import ArticleEditor from '@/components/ArticleEditor.vue'
import { onBeforeMount, ref } from 'vue'
import { onBeforeRouteUpdate, useRoute } from 'vue-router'

const createMode = ref<boolean>(false)
const editMode = ref<boolean>(false)
const article = ref<ArticleDocumentResponse>()

const route = useRoute()

async function resolveView() {
  if (route.path == '/create-article') {
    createMode.value = true
    console.log('文章创建模式')
  } else {
    const articleResponse = await DefaultService.readArticleArticlesArticleIdGet(
      route.params.id.toString()
    )
    article.value = articleResponse.article
    editMode.value = true
    console.log('文章编辑模式')
    console.log(article)
  }
}

onBeforeMount(() => {
  resolveView()
})

onBeforeRouteUpdate(() => {
  resolveView()
})
</script>

<template>
  <q-page padding>
    <ArticleEditor v-if="createMode" create-mode />
    <ArticleEditor v-if="editMode" edit-mode :article-to-edit="article" />
  </q-page>
</template>

```

我们重点关注的其中的组件`<ArticleEditor>`，它是一个*单文件组件（SFC）*，单独写在一个名为`ArticleEditor.vue`的文件里，当这个组件按照Vue的**开发规范正确书写并编译**后，会由`@vue/compiler-sfc`进行编译，成为一个**标准的JavaScript（ES）板块**，因此就可以使用原生JS的导入与导出语法。

但是更重要的是，当在一个组件中导入了另一个组件的时候，他们之间就称为父子组件的关系，实践中很多场景是需要组件之间的*模板嵌套、数据传递、样式复用等*【*对应单文件组件的三大组成部分*】的，因此组件（在当前的一般实践中就指的是单文件组件SFC）就是Vue框架提出的最重要的抽象——关系到==代码组织逻辑==。

## 组件的注册——模板之间的嵌套
一个 Vue 组件在使用前需要先被“注册”，这样 Vue 才能在渲染模板时找到其对应的实现。组件注册有两种方式：**全局注册**和**局部注册**。
### 全局注册
我们可以使用**Vue应用实例**（`createApp` 函数创建的应用实例）的`.component`方法，让组件在当前 Vue 应用中全局可用：
```js
import { createApp } from 'vue'
import MyComponent from '../components/ComponentB.vue'

const app = createApp({})

// 组件名与组件实现不分离的情况
app.component(
  // 注册的名字
  'ComponentA',
  // 组件A的实现
  {
    /* ... */
  }
)
// SFC的情况
app.component('ComponentB', ComponentB)
```
全局注册虽然很方便，但有以下几个**问题**：
1. 全局注册，但并没有被使用的组件==无法在生产打包时被自动移除== (也叫“tree-shaking”)。如果你全局注册了一个组件，即使它并没有被实际使用，它仍然会出现在打包后的 JS 文件中。
2. 全局注册在大型项目中使==项目的依赖关系==变得不那么明确。在父组件中使用子组件时，不太容易定位子组件的实现。和使用过多的全局变量一样，这可能会影响应用长期的可维护性。

相比之下，局部注册的组件*需要在使用它的父组件中显式导入，并且只能在该父组件中使用*。它的优点是==使组件之间的依赖关系更加明确，并且对 tree-shaking 更加友好==。
### 局部注册
在使用 `<script setup>` 的单文件组件中，导入的组件可以直接在模板中使用，无需注册：
```vue
<script setup>
import ComponentA from './ComponentA.vue'
</script>

<template>
  <ComponentA />
</template>
```
如果没有使用 `<script setup>`，则需要使用 `components` 选项来显式注册：
```js
import ComponentA from './ComponentA.js'

export default {
  components: {
    ComponentA
  },
  setup() {
    // ...
  }
}
```
对于每个 `components` 对象里的属性，它们的 key 名就是注册的组件名，而值就是相应组件的实现。上面的例子中使用的是 ES2015 的缩写语法，等价于：
```js
export default {
  components: {
    ComponentA: ComponentA
  }
  // ...
}
```
请注意：局部注册的组件在后代组件中**不**可用。在这个例子中，`ComponentA` 注册后仅在当前组件可用，而在任何的子组件或更深层的子组件中都不可用。
### 命名规范
必须要使用`PascalCase` 作为组件名的注册格式，这是因为：
1. PascalCase 是合法的 JavaScript 标识符。这使得在 JavaScript 中导入和注册组件都很容易，同时 IDE 也能提供较好的自动补全。
2. `<PascalCase />` 在模板中更明显地表明了这是一个 Vue 组件，而不是原生 HTML 元素。同时也能够将 Vue 组件和自定义元素 (web components) 区分开来。

| 命名法             | 示例             | 规则特点               |
| --------------- | -------------- | ------------------ |
| **PascalCase**  | `MyComponent`  | 每个单词首字母均大写，无分隔符    |
| **camelCase**   | `myComponent`  | 首单词小写，后续单词首字母大写    |
| **kebab-case**  | `my-component` | 全小写，单词间用连字符 `-` 分隔 |
| **snake\_case** | `my_component` | 全小写，单词间用下划线 `_` 分隔 |

## `Props`——父子组件之间传递属性的桥梁

通俗地说，**Props 就是父组件给子组件"传参"的方式**。就像调用函数时传入参数一样，父组件在子组件的标签上通过属性绑定数据，子组件内部声明自己接收哪些数据，这样就完成了一次父子通信。
### `Props`的声明
一个组件需要**显式声明**它接受哪些 props，这样 Vue 才能区分"这是外部传入的数据"和"这是普通的 HTML 属性"。
在 `<script setup>` 中，使用 `defineProps()` 宏来声明：
```vue
<script setup>
// 数组形式：只声明名称
const props = defineProps(['foo'])
console.log(props.foo)

// 对象形式：声明名称 + 类型
defineProps({
  title: String,
  likes: Number
})
</script>
```
也有`Optional API`的代码风格：
```js
export default {
  props: ['foo'],        // 数组形式
  // 或
  props: {
    title: String,
    likes: Number
  },
  setup(props) {
    console.log(props.foo)
  }
}
```
对象形式的声明有两个好处：一是相当于**组件的自文档化**，别人一看就知道该传什么类型；二是如果传错了类型，Vue 会在控制台抛出警告，帮你提前发现 bug。

如果搭配 TypeScript 使用，还可以用**类型标注**的方式声明，更加简洁：
```vue
<script setup lang="ts">
defineProps<{
  title?: string
  likes?: number
}>()
</script>
```

### 响应式`Props`解构
在 Vue 3.5 之前，如果你这样写：
```js
const { foo } = defineProps(['foo'])
```

解构出来的 `foo` 是一个**常量**，会丢失响应性——父组件更新了，子组件里解构的变量不会跟着变。
从 Vue 3.5 开始，编译器会自动帮你处理：在同一个 `<script setup>` 代码块中访问解构变量时，Vue 会自动将其转换为 `props.foo`，因此**解构后的变量仍然保持响应性**。
此外，3.5+ 支持用 JavaScript 原生语法给解构的 prop 设默认值：
```ts
const { foo = 'hello' } = defineProps<{ foo?: string }>()
```

> **注意**：如果你把解构的 prop 传给 `watch()` 等函数，需要包在 getter 里，例如 `watch(() => foo, ...)`，否则传递的是一个静态值而非响应式数据源。

### 传递 Prop 的细节
#### 命名格式：声明用 camelCase，传递用 kebab-case
在 JavaScript 里，prop 名用 **camelCase**（如 `greetingMessage`），因为它们是合法的标识符，可以直接在表达式中使用：
```vue
<span>{{ greetingMessage }}</span>
```

但在模板中传递时，为了贴近 HTML 的书写习惯，推荐写成 **kebab-case**：
```vue
<MyComponent greeting-message="hello" />
```

#### 静态 vs 动态
直接写死字符串，是**静态 prop**：
```vue
<BlogPost title="My journey with Vue" />
```

需要绑定变量或表达式时，用 `v-bind`（简写 `:`），这是**动态 prop**：
```vue
<!-- 绑定变量 -->
<BlogPost :title="post.title" />

<!-- 绑定表达式 -->
<BlogPost :title="post.title + ' by ' + post.author.name" />
```

#### 传递不同的值类型
**任何类型**的值都可以作为 prop 传递，但要注意：只要不是纯字符串，就需要用 `v-bind` 告诉 Vue "这是一个 JavaScript 表达式"：
```vue
<!-- Number -->
<BlogPost :likes="42" />

<!-- Boolean：只写属性名不传值，会隐式转为 true -->
<BlogPost is-published />

<!-- 显式传 false，必须用 v-bind -->
<BlogPost :is-published="false" />

<!-- Array / Object -->
<BlogPost :comment-ids="[234, 266, 273]" />
<BlogPost :author="{ name: 'Veronica', company: 'Veridian Dynamics' }" />
```

#### 用一个对象绑定多个 prop
如果你想把一个对象的所有属性都作为 props 传入，可以用**没有参数的 `v-bind`**：
```vue
<BlogPost v-bind="post" />
```

这等价于：
```vue
<BlogPost :id="post.id" :title="post.title" />
```

### 单向数据流
这是 Props 最核心的原则：**props 是单向绑定的**。父组件的数据变化会向下流向子组件，但子组件**不能**直接修改 prop。如果你尝试 `props.foo = 'bar'`，Vue 会在控制台抛出警告。
> 为什么这样设计？因为一旦允许子组件随意修改父组件的数据，整个应用的数据流就会变得混乱，难以追踪 bug。

实际开发中，你想修改 prop 通常出于两种需求，都有标准解法：
**场景 1：prop 只是初始值，子组件想自己维护后续状态**

不要直接改 prop，而是**新定义一个本地数据属性**，把 prop 作为初始值：
```vue
<script setup>
const props = defineProps(['initialCounter'])
const counter = ref(props.initialCounter)  // 从此与 props 脱钩
</script>
```

**场景 2：需要对 prop 做转换（如格式化、大小写转换）**
使用**计算属性**，它会自动追踪 prop 的变化：
```vue
<script setup>
const props = defineProps(['size'])
const normalizedSize = computed(() => props.size.trim().toLowerCase())
</script>
```

**关于对象/数组类型的特殊提醒**

对象和数组是按**引用**传递的。虽然你不能替换整个 prop（`props.arr = []` 会报错），但**可以**修改其内部值（`props.arr.push(1)` 不会报错）。然而这会导致子组件"偷偷"改了父组件的数据，数据流变得不透明。**最佳实践是：子组件通过抛出事件通知父组件去改**。

### Prop 校验
你可以对 props 进行更细致的校验，这在开发公共组件库时特别有用：
```js
defineProps({
  // 基础类型检查
  propA: Number,

  // 多种可能的类型
  propB: [String, Number],

  // 必传字段
  propC: {
    type: String,
    required: true
  },

  // 必传，但值可以为 null
  propD: {
    type: [String, null],
    required: true
  },

  // 带默认值
  propE: {
    type: Number,
    default: 100
  },

  // 对象/数组的默认值必须用工厂函数返回
  propF: {
    type: Object,
    default(rawProps) {
      return { message: 'hello' }
    }
  },

  // 自定义校验函数
  propG: {
    validator(value, props) {
      return ['success', 'warning', 'danger'].includes(value)
    }
  }
})
```

一些补充规则：
- 所有 prop **默认都是可选的**，除非声明 `required: true`。
- 除 `Boolean` 外，未传递的可选 prop 默认值为 `undefined`。
- `Boolean` 类型的未传递 prop 会被转换为 `false`（可以通过设置 `default: undefined` 来改变这一行为）。
- 如果声明了 `default`，那么无论 prop 是未传递还是显式传了 `undefined`，都会使用默认值。

**运行时类型检查支持的原生构造函数**：`String`、`Number`、`Boolean`、`Array`、`Object`、`Date`、`Function`、`Symbol`、`Error`。也可以传入自定义类，Vue 会用 `instanceof` 进行校验。

### Boolean 类型的特殊转换规则
为了贴近原生 HTML 的 boolean attributes 行为，声明为 `Boolean` 类型的 prop 有特殊的转换规则：
```vue
<!-- 只写属性名，等同于传入 :disabled="true" -->
<MyComponent disabled />

<!-- 不写该属性，等同于传入 :disabled="false" -->
<MyComponent />
```

当一个 prop 允许多种类型时，**`Boolean` 的转换规则是否生效，取决于它在类型数组中的位置**：
```js
// Boolean 在前 → 适用转换规则（disabled 不写就是 false）
defineProps({ disabled: [Boolean, String] })

// Boolean 在后 → 不适用转换规则（disabled 不写会变成空字符串 ""）
defineProps({ disabled: [String, Boolean] })
```
