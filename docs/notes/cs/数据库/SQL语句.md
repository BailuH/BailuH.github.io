# SQL语句的生命周期
	*以下内容来源于Gemini，本人才疏学浅，此时知识尚不能判别真伪，仅供参考*

如果我们模仿这个视角，追踪一条 SQL 语句：`UPDATE items SET price = 100 WHERE id = 1;` 在 MySQL（InnoDB 引擎）中的“生命历程”，我们可以将其称为 **“一条 SQL 的磁盘逆袭之旅”**。

---
### 第一站：从网络协议栈到进程上下文

当你在终端按下回车，这串字符并不是直接跑在 CPU 上，而是被封装成 **MySQL Client 协议包**。
- **传输层：** 通过 TCP/IP 到达服务器 3306 端口。
- **应用层：** MySQL 的“连接池”接收到请求，验证你的权限。
- **上下文切换：** OS 将此请求分配给一个 MySQL 线程。此时，这串字符串存放在该线程的**私有堆栈**内存中。
---
### 第二站：SQL 引擎的“大脑”加工（编译与执行计划）

这部分对应 CSAPP 里的“从源代码到可执行文件”的过程，只不过 SQL 是实时编译的。

1. **解析（Parser）：** 词法分析器把字符串切碎，识别出 `UPDATE` 是指令，`items` 是表。生成一颗 **AST（抽象语法树）**。
2. **预处理（Preprocessor）：** 检查表名、列名是否存在。
3. **优化器（Optimizer）：** 这是最关键的一步。它会查看 `id` 是否有索引。
    - 如果有 B+ 树索引，它决定走 **Index Scan**。
    - 如果没有，它只能走 **Table Scan**（全表扫描）。
    - **产物：** 生成一串类似汇编指令的“算子序列”。
---
### 第三站：存储引擎与内存层次（Buffer Pool 的魔法）

这里是 DBMS 与 OS/硬件协作最紧密的地方。

1. **内存查找：** 执行器问 InnoDB 引擎：“给我 id=1 的这一行”。InnoDB 首先去 **Buffer Pool（缓冲池）** 查找。
2. **磁盘预取：** 如果内存里没有，InnoDB 会触发一个 **Page Fault（逻辑上的）**，请求 OS 读取磁盘。
    - **底层：** 调用 `read()` 系统调用。OS 调度驱动程序，磁头移动，将 16KB 的数据页（Page）读入内存。
3. **内存修改：** 既然你不需要并发考虑，InnoDB 直接在内存中找到那一行，修改 `price` 列的值。
    - **状态改变：** 此时，内存里的这个 Page 变成了 **Dirty Page（脏页）**。
---
### 第四站：持久化的信仰（WAL 机制）

按照 CSAPP 的逻辑，如果此时断电，内存数据就丢了。但数据库保证“只要提交成功，数据永不丢失”。

1. **生成 Redo Log：** InnoDB 不会立刻把脏页写回磁盘（太慢了），而是写一条物理日志：_“在某文件某偏移量处，把值改成了 100”_。
2. **磁盘同步（fsync）：** 这条日志被写入 **Redo Log Buffer**，然后调用 `fsync`。
    - **关键点：** 这是 DBMS “强迫”操作系统跨过内核缓冲区，直接把电流刷入 SSD/HDD 的闪存介质中。
3. **确认提交：** 只要日志落盘，MySQL 就会告诉你 `Query OK`。即使一秒后停电，重启后 MySQL 也能根据日志恢复。
---
### 第五站：异步的终局（Checkpoint）

SQL 脚本已经执行完了，但对于计算机系统来说，这事儿没完：

- **后台线程：** 在系统负载较低时，MySQL 的后台线程会悄悄启动。
- **脏页刷盘：** 它将 Buffer Pool 中那些被修改过的脏页，通过 OS 的 `write()` 接口，真正写回到磁盘上的 `.ibd` 数据文件中。
- **清理日志：** 一旦数据页安全落地，原本的 Redo Log 就可以被覆盖重用了。
---
### 总结：SQL 脚本的形态演变

1. **形态 A (文本)：** `UPDATE...`（存储在内存缓冲区字符串）。
2. **形态 B (逻辑结构)：** 优化器里的 **查询树/算子流**（C++ 对象）。
3. **形态 C (内存映像)：** Buffer Pool 里的 **二进制数据页**（16KB 的数据块）。
4. **形态 D (持久化流)：** 磁盘上的 **Redo Log 字节流**（顺序写入）。
5. **形态 E (最终归宿)：** 磁盘 `.ibd` 文件中的 **B+ 树节点**。

# 数据查询

## 零、SQL语句的执行顺序
	这个执行顺序是由SQL引擎决定的。必须熟练掌握这个执行顺序，才能写出逻辑不出错的SQL语句。

| **顺序**       | **关键字**    | **功能**               |
| ------------ | ---------- | -------------------- |
| 1 (FROM)     | `FROM`     | 找到数据在哪张表             |
| 2 (ON/JOIN)  | `JOIN`     | 如果涉及多张表，先连起来         |
| 3 (WHERE)    | `WHERE`    | 初步过滤（不能跟聚合函数）        |
| 4 (GROUP BY) | `GROUP BY` | 分组（如：按部门统计）          |
| 5 (HAVING)   | `HAVING`   | 分组后的二次过滤             |
| 6 (SELECT)   | `SELECT`   | 挑选出你最终想要的列           |
| 7 (ORDER BY) | `ORDER BY` | 排序（升序 ASC 或 降序 DESC） |
| 8 (LIMIT)    | `LIMIT`    | 限制输出行数（如：前 10 名）     |

## 一、基础查询

>这个部分主要是要掌握对于数据库原始数据的取用，核心目标就是要做到指哪打哪，需要什么数据就拿来什么数据，不多不少。

### 1. SELECT后面能跟什么？
- 最常见的是跟已有关系（表）的属性列。特殊地，可以用==通配符==\*来表示全部属性列。
- 还可以跟一些表达式，但是要注意，如果采用这种方式查找的数据就不是原始数据（raw-data）了，变化的方式就取决于表达式的形式。
	例如，
	算数表达式，2014-Sage（Sage属性得是一个整数可以正常参与算数运算）；
	字符串常量（像”回声“一样，每个元组对应的分量均为这个字符串常量，列名不同方言可能不同）；
	\*逻辑表达式（MySQL用0/1代表逻辑值，是可以支持的，但是有的方言可能不行，得换用case表达式）；
	还可以跟函数（例如lower(Sdept)，要求属性Sdept是字符串，能将其内容转为小写)
- ==ALL与DISTINCT关键字==：对查找的元组是否进行去重处理，默认是ALL，即不去重处理。

### 2. 利用WHERE子句实现条件筛选
值得注意的是，where子句的作用在于筛选“行”即符合条件的元组，不改变原始数据，根据需要只要记住一些重要的谓词即可。

| 查询条件 |                   谓词                    |
| ---- | :-------------------------------------: |
| 比较   | =,>,<,>=,<=,!=,<>(更加推荐的写法，表示不等于),!>,!<; |
| 确定范围 |               BETWEEN AND               |
| 判断集合 |                   IN                    |
| 模糊匹配 |                  LIKE                   |
| 空值   |                 IS NULL                 |
| 逻辑运算 |   AND，OR(用于连接两个或多个条件)，NOT（用于附加在上述的条件）   |
	只强调几个容易犯错或者不好理解的点：
1. 关于空值（NULL）：在数据库中，任何数据与NULL进行比较都不会返回逻辑值**TRUE/FALSE**，而是返回**UNKNOWN**，因此在处理有关空值的条件的时候，数据库的写法就是鼓励单独处理，也就是用IS NULL来处理
2. 关于模糊匹配：注意只能用于字符串类型的数据（至少这一点很像正则表达式），记住两个通配符的作用。（a%b和a_b区别？）
- % 表示任意长度（可以为0）的字符串
- __  表示任意单个字符

### 3. ORDER BY子句进行“查询结果的排序”
	ORDER BY子句可以对查询结果按照一个或多个属性列的升序（ASC）或者降序（DESC）排列，默认是为升序（ASC）。
仍然是两个值得注意的点。
- 空值的排序是由具体的系统实现决定，只需要前后一致就好。
- 这里的列表应当是有序的，多个属性列的情况，谁在前就先按照谁进行排序。
## 二、聚合与分组

>这个部分在单表、多表都能用，多表连接后其实也就可以看成单表，因此只需要搞明白单表的聚合和分组是怎么做的就好，这部分非常重要，因为可以查询**统计数据**，具有很高的实用和业务价值。
### 1. 先说聚合函数
	前面说过，SELECT语句后面可以“紧跟”一个函数，我们最常见的就是跟一个聚合函数，但是要注意的一点是，聚合函数里只能对一个属性列进行操作，当然也可以是某种表达式（几乎就和SELECT后面能跟的东西一样多）

| 形式       | 功能               |
| -------- | ---------------- |
| COUNT(*) | 统计元组个数           |
| COUNT    | 统计某一属性列中非空的值的个数  |
| SUM      | 统计某一属性列的总和（数值型）  |
| AVG      | 统计某一属性列的平均值（数值型） |
| MAX      | 统计某一属性列的最大值      |
| MIN      | 统计某一属性列的最小值      |
还是有几个容易出错或值得注意的点：
1. 除了count(\*)以外，其余的聚合函数遇到NULL都会==跳过==，完全不处理，这又和我们前面所提到的空值处理原则相联系了。
2. count(\*)就连整个元组均为NULL也能正常统计，这意味着count(\*)并非是在“找有没有非NULL”，而是就在统计元组个数。
3. 聚合函数的参数前可以加==DISTINCT/ALL==关键字，决定进行聚合之前要不要**先进行**元组的去重处理，默认是ALL。
4. 聚合函数不可以写在WHERE子句，只能写在SELECT和HAVING之后。详情可以参考后续关于HAVING和WHERE的对比。

### 2. 再来看看分组
	前面提过，聚合函数是只能接受一个参数的，如果你需要涉及对多个属性列的聚合，那么就得用到分组了。一般情况下标准的SQL都有一个“ONLY_FULL_GROUP_BY"模式，要求在没有GROUP BY子句的时候，SELECT后紧跟的对象列表不能包含任何非聚合列（即没有使用聚合函数的属性列）

#### 分组原理
GROUP BY子句将查询结果按照一列或多列的值进行分组，只有每一列的值均对应相等的才能分为一组。

#### 与SELECT的联系
关于前面的“ONLY_FULL_GROUP_BY"模式，如果有GROUP BY子句的话，那么SELECT就可以跟**由分组依据的属性列所决定的属性列**，换句话说查询列表的属性必须**函数依赖于**分组属性。

下面这个查询语句是不合法的：
```sql
select Sno,grade,count(*)  
from sc  
group by Sno;
```
由于grade属性并不是由Sno所**决定**，因此这个会报错，可以修改如下：
 ```sql
select Sno,Cno,grade,count(*)  
from sc  
group by Sno,Cno;
 ```
在我们所定义的SC表中，grade由Sno和Cno共同决定，那么分组属性列表为Cno,Sno时，grade就可以写进SELECT查询列表内。

#### 与HAVING的联系
HAVING子句的执行顺序位于GROUP BY和SELECT之间，很好理解它的功能就是在分组后可以再一次筛选，筛选的对象就是GROUP BY所分的组。

对于这样一个查询语句，它的功能是查询每个同学总共选了几门课：
```sql
select Sno,count(Cno)  
from sc  
group by Sno;
```
结果是这样的：

如果我们想知道选课门数大于两门的同学，可以利用HAVING子句筛选掉不符合的**分组**。
```sql
select Sno,count(Cno)  
from sc  
group by Sno  
having count(cno)>2;
```
结果就是这样，可以看到选课门数为2的同学被筛选掉了。

#### ❓HAVING与WHERE的区别
>HAVING和WHERE都是用于按照条件进行筛选，他们的作用却不相同，也不是很好区分。
	从原理上说，二者的作用对象是不同的，WHERE子句的作用对象是表或者视图，它的“筛选粒度”可以是一个单一元组；HAVING子句的作用对象是经过GROUP BY之后的分组，它的“筛选粒度”是一个“分组”（尽管这个分组有可能只有一个元组）
	有一个实践中好区分的方法，也算是他们的特点是，WHERE是不可以用聚合函数作为条件来筛选的，相反HAVING的条件往往都会使用聚合函数来构造条件。原因在于，经过GROUP BY子句后，HAVING所能筛选的视图中的属性只包含分组列和聚合函数构造的列了。

## 多表查询——连接与嵌套
