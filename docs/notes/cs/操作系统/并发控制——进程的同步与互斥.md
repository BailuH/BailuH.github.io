> 简单来说，为什么要有并发控制的原因在于，在操作系统中我们需要这样一类进程——它们共享某一些数据，共同完成某些任务，彼此之间的运行过程是相互影响的。因此，**对共享数据的并发访问可能会造成数据的不一致性**，这就引出了进程互斥的问题；**有的进程的运行依赖于其他进程对共享数据的操作，各个进程需要保持一定的时序来运行**，这就引出了进程同步问题。

# 关键概念归纳
## 1.协作进程
- 协作进程（cooperating process）是进程并发控制的主要研究对象。与之相对应的概念称为独立进程（independent process）。
- 协作进程指的是那些会在执行过程中影响别的进程或者受其他进程影响的进程。它是一个对偶概念（即协程总是成对出现）
- 协程的一大特征就是数据的共享，包含两种方式：
1. 直接共享逻辑地址空间，也就是说协程之间共享==代码和数据==
2. 仅通过文件或消息来进行数据的共享
## 2.竞态条件
	如果多个线程/进程同时操作（访问并修改）同一个共享资源，而最终结果依赖于它们执行的先后顺序，并且这种顺序无法预测或者控制的时候，就发生了竞态条件（race condition）。
### 根本原因：
CPU调度的中断机制以及抢占式调度。导致重要的代码（访问共享资源的代码，也就是后面要介绍的临界区）不是原子执行的。
## 3.临界区
- 临界区指的是程序中需要访问*共享资源*（包含代码、数据和硬件）的那段代码。
## 4.自旋锁与非自旋锁——是否忙等（busy-waiting）
>这个概念的引出主要是为了说明这样一个问题：当<u>中断发生在某个进程或线程释放锁之前</u>，别的进程同样想访问临界资源的时候，这个进程是一直忙等待，占有CPU但不进行任何操作呢；还是被阻塞，释放CPU进而用来作更有价值的事情呢？

从效率性的角度来说，我们肯定都希望我们提供的并发控制机制不要忙等待，因为这样会降低CPU的利用率，因此除了Peterson算法这类纯依靠软件实现且使用范围狭窄的情况，我们在设计操作系统并发控制的时候，总是倾向于**利用硬件**，来==保证某些操作的原子性==，从而打破静态条件的根本魔咒——不合时宜的中断与抢占式调度。
# 进程互斥
>为了更好理解和评价我们引入的机制的效果，我们最好要有一套对于进程互斥的评价标准：
>1. 正确性：要实现进程互斥的目的
>2. 公平性：当一个进程放弃CPU的时候，要确保协作进程的每个进程都有平等的机会占有CPU
>3. 效率性：可以从CPU的利用率出发，分析单CPU、多CPU的不同情况。

## 2. 信号量——操作系统内核的介入
### 原理
#### 信号量的基本思想
- 信号量是一个为协作进程所**共享**的一个整型变量，当S>0时，信号量S表示可供协作进程（并发进程）使用的共享资源数量；当S<0时，|S|表示正在使用资源的进程或线程的数目。
- 除了赋初值以外，信号量的改变只能用PV操作来进行，并且PV操作是原语。（具有原子性，即不可中断）
P原语的伪代码描述如下：
```c
P(Semaphore S){
while(s<=0)
;//不进行任何操作
s--;//要注意这个操作实际上应当是原子操作
}
```
V原语的伪代码描述如下：
```c
V(Semaphore S){
s++;
}
```
#### 二值信号量/公有信号量/互斥信号量（信号量用作锁）
并发编程使用信号量控制的代码大致长这样，这是一个进程互斥的简单示意：
```c
#include<semaphore.h>
sem_t s;
sem_init(&s,0,1);//将信号量的初始值设为1
void process_A(){
...
P(&s);
//临界区代码
V(&s);
...
}

void process_B(){
...
P(&s);
//临界区代码
V(&s);
...
}
```
#### 私有信号量/资源信号量（信号量用作条件变量）
这是一个进程同步的简单示意：
```c
#include<semaphore.h>
#include<pthread.h>

sem_t s;
void child(){
	printf("child\n");
	V(&s);
}

int main(){
	sem_init(%s,0,0)//初始值应当是0
	printf("parent:begin\n");
	pthread_t c;
	Pthread_create(c,NULL,child,NULL);//创建子进程
	P(&s);//等待子进程结束
	printf("parent:end\n");
	return 0;
	}
```
>🤔❓**信号量可以充当锁和条件变量的泛化吗**
>信号量确实简单易用，而且可以实现锁和条件变量能解决的一些问题，因此确实能算作锁和条件变量的一种泛化。但考虑到利用信号量去实现条件变量的难度，其实这样的一种泛化并没有我们想象的那么通用。更好的做法（也是现代操作系统的做法）是两者兼用。
### 实现：以POSIX标准为例
	关于信号量的实现方式，按照迪杰斯特拉的构想需要两个原子性的原语，即P操作和V操作，至于这两个原语的具体实现，则是由操作系统内核人员根据硬件提供的指令集编写出合适的系统调用（System call）封装成合适的用户接口（User API）来让程序员“利用软件方式”来实现。

信号量的实现方式在类UNIX的实现方式和迪杰斯特拉定义的信号量有一些细微的差别，这里的信号量并没有实现在信号量的值为负数的时候，让它反映处等待的线程数。事实上，这个值永远不会小于0，这样的行为更容易实现。
