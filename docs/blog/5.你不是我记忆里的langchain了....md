# 我与Agent开发斗争的这半年
（封面是北京昌平的雪）

>事情是这样的，最近我一直在写自己规划已久的一个项目，是关于LangGraph的。现在是寒假，学校没什么人，我难得有此机会静下心来去写代码，这段时间的生活已经和代码和LangChain和LangGraph的文档深度绑定了。（其实一边值班，一边写代码摸鱼的生活也挺好的，希望未来毕业我能找到这样一份工作）
>
>话说远了，事情是这样的...

![](https://bailu-1382509292.cos.ap-beijing.myqcloud.com/obsidian/20260120181834004.jpg)

是的，LangChain有了一次很大的变化。但在这里我并不想深入地谈它的技术变革，只是有感而发，因此决定写下这些文字。这更像是一篇关于我如何逐渐修正自己对 Agent 理解的记录，而不是一篇技术教程。

## 我和LangChain第一次见面
其实也没有很久，大概也就是半年前，暑假，我开始构思现在正在写的项目的时候。
当时一直想做一个基于**多智能体框架**的模拟法庭系统，于是在互联网的各个角落（其实主要就是AI）搜寻合适的技术框架准备自学，于是就搜到了LangGraph，于是开启了轰轰烈烈的学习。（写到这又想到好多小细节，连着热点看网课、许下小小心愿要写下自己的学习过程，于是有了这个公众号）
但其实我当时在暑假主要是学了LangGraph，只是因为写代码的时候还是会用到LangChain的一些小组件，但其实并没有怎么钻研过。因为相比LangChain，LangGraph似乎更符合我的构想，尽管当时理解地也不深刻，许多细节当时想的也不对，但是误打误撞技术选型还是选对了。
当时做出选择主要是基于这样一个想法：LangChain构建代理的方式更接近一个线性的工作流，其中不支持循环流转等方式，自然是不适合法庭辩论这种短兵相接、你来我往的流程。于是，==“LangChain = 工作流”==的想法就一直留存在我的脑海里，直到今日。

## 原本的LangChain到底是什么样的？
这个部分可能涉及一些技术内容，但是都是一些比较好理解的、也不是什么定论的东西，相信没有计算机背景的朋友们也能理解。
后来开学了，对于LangGraph的学习虽然不算大成，但是也是小有所得。主要是对于LangGraph的设计哲学谙熟于心，并一直为之感到绝妙，有一段时间我认为LangGraph就是开发多智能体框架的最佳选择。
在学校的课业压力还是比较大的，而且我也逐渐投身到为我的项目招揽人才的社交活动中，于是也几乎没怎么推进LangGraph学习。但是在校期间，我参与了几个小的社会赛事，有律所举办的，有高校举办的，关于Agent的赛事（不得不说国内还是不论是学界还是社会还是一直在努力追热点），期间也接触过几个其他的agent开发平台，主要是国内平台，毕竟有赛事要求。例如腾讯云智能体开发平台、Coze、Dify。
对于这些平台的印象都没有特别好，当然也有我没有充分玩转平台的原因（求生欲）。最大的印象是，这些平台都长得很“工作流”，所谓“低代码”的抽象，怎么似乎都只走了工作流这条老路呢？这些平台在国内也都算是小有名声吧，其中也不乏将自己的“工作流”作为核心卖点并受到追捧的。每当这个时候我的脑海都会闪过LangChain的身影，一样的线性工作流。虽然我在这段时间没有进一步学习这两个框架了，但是还是时不时会追一些热点，也一直在尝试理解Agent这个概念（甚至还尝试写了一篇文章，尽管现在看来是有一些谬误的——）。因为我第一次正式地接触Agent这个概念就是LangChain带给我的——LangGraph我一直将其理解为**专门**构建多智能体应用的框架。又因为前文说过的==“LangChain = 工作流”==的想法一直挥之不去，于是导致我很长时间一直觉得Agent就是要搭一个工作流指挥AI干活，加上接触过的几个低代码平台无一不采纳了工作流这种方式，更是让这种印象根深蒂固...

## 所以Agent到底是什么？
**我希望我这次能好好地、至少不出错地回答这个问题**。但在那之前，我想接着先说说后来的一件事，就是我在Github上发现了这个项目。（为了尊重作者，这里贴上原仓库的URL：https://github.com/ruc-datalab/DeepAnalyze）
![image.png](https://bailu-1382509292.cos.ap-beijing.myqcloud.com/obsidian/20260120190913437.png)
我现在还记得当时我的脑子里想的是什么——“Agentic LLM？LLM就LLM，你不应该是工作流里面那个，被人配置的模型吗，怎么还碰上瓷了，非要说自己是个Agent”。于是我就很好笑地继续往下滑，试图在README.md中寻找任何工作流的蛛丝马迹，然后看到类似于“本项目已经将模型开源到hugging face上，模型参数和权重都可以下载”的时候，我脑海对于Agent的认知出现了巨大的裂痕。**也是从这里开始，我第一次意识到：我此前对 Agent 的理解，可能从根子上就偏向了“编排”，而不是“自主”。**
后来我和GPT聊了很久（已经找不到记录了，故不贴出），最终它给出的一个结论暂时结束了这个事件——LLM本身是可以再构建一层“代理层”的，似乎是以不同于模型参数的方式在影响着LLM的行为。（我对深度学习的专业知识并不了解，因此我也无法verify这个说法，如果有读者朋友懂的话欢迎留下评论，感激不尽）。
但至少这让我明白了一件事，就是Agent确确实实不等于搭一个工作流让AI去干活，事实证明这也不是构建Agent的一个很好的方式。现在我希望我能给Agent下一个够用的定义：**Agent是一个能够自主规划其思考与行动路径的，基于LLM的一个复杂系统**。

| Agent的特征 | 市面的例子                                 |
| -------- | ------------------------------------- |
| 自主规划思考   | tool calling、联网搜索、skills...           |
| 自主规划行动   | 各种垂类智能体，例如数据分析智能体、舆情分析智能体、code agent等 |
不论是传统的所谓tool calling、联网搜索，到现在比较火的“skills”，其实都可以归入“自主规划思考路径”的范畴；至于“自主规划行动路径”，就可以理解为各种垂类大模型实现的各种复杂的功能，例如上文仓库中的数据分析、法律分析、开发者使用的各种AI开发工具（claude code、codex、cursor等）均在此列；它们都是LLM驱动的，这里的LLM就指的是比较狭义的LLM，也符合我以前写的文章的定义，就是一个庞大的深度学习网络，给定输入，根据算法和概率产生回复的那个“LLM”。
## 工作流构建Agent为什么不好用？
半年时间，说长不长，说短也不短。半年足够Agent技术换一片天了，不论是在Coze这类低代码平台，还是在LangChain这样的纯代码的框架，==工作流（线性编排节点）的方式，至少在需要自主规划与动态决策的 Agent 场景中，似乎已经悄悄地被扫进历史的垃圾堆了。==为什么工作流这么难做出什么真正有价值的产品呢？个人看法是，工作流这种方式束缚了LLM的潜力，让**大模型“戴着镣铐跳舞”**，何以见得？
我在这学期干过一件非常蠢的事情，现在想起来还在庆幸最后没有再投入很多的时间去做这个事。就是去Coze拿工作流搭建一个类似写作助手的东西，主要是为了敷衍大学的平时作业。（不好的行为，本人深刻反省，大家不要这么做！）
![image.png](https://bailu-1382509292.cos.ap-beijing.myqcloud.com/obsidian/20260120194508543.png)
当时多半是被Coze的繁多功能给绕的不知天地为何物了（并非指责扣子，我自己用的方式不好）。试过的都知道，这种工作流弄出来的文章，可能还不如你直接点进GPT或者DS的官网直接生成来的强......究其原因还是，我前面说的，工作流就像是”戴着镣铐跳舞“，原本大模型接受你的输入，基于本身算法具有的概率性（以及很多后来的研究成果），大模型自身是有一定创造性的（你不得不承认这点）。Agent的引入是为了让这份创造性更可控，让用户更好地利用这份创造性，将简单的”创造“变为有条理和秩序的”规划“。但是事实上，大家以我搭建的工作流为例来看，这些每个单独的节点目前的大模型都可以单独做到，不论是复杂的插件还是读写文件，搜索网页等，你人为地编排这些节点，其实无意中你就将自己解决这套问题的思路给强加给了AI，让AI只能顺着你的思路去执行任务，某些情境下可能有点用，但大多数时候其实都没法起到1+1>2的价值（人类经验+AI效率）。

## 戴着镣铐跳舞——LangChain为什么变了？
前文用“戴着镣铐跳舞”来形容工作流这种设计Agent的模式，其实多少有些“望文生义”之嫌。“戴着镣铐跳舞”本来是被用来比喻诗歌创作，<u>最早出自美国批评家布里斯·佩里，原话大意是：“（诗人）乐意戴着脚镣跳舞，并且要戴个别诗人诗人的脚镣”，</u>后来我国新月派的诗人以此用来强调“格律”是诗歌创作不可或缺的因素。其实这个比喻用来形容Agent的设计哲学我觉着是合适的，**Agent就是应当让大模型戴着脚镣跳舞**，不然完全不可控，无法上生产，无法创造额外价值，最后也只能被当作高级点的玩具罢了。
工作流这种方式正如前文所说，虽然给大模型戴上了镣铐，但是也给大模型砍断了手足，再也无法依照自己的意志去跳舞了，问题从来不在于“要不要镣铐”，而在于镣铐是否允许舞者自己选择舞步。
那变化后的LangChain又是什么样呢？它和当下爆火的“skills”有没有设计的共通之处呢？
接下来的内容可能涉及一些技术层面的解释，但我会尽量说的比较简单。首先来看看现在的LangChain文档，也是我今日突然发现LangChain变化的缘由。
![](https://bailu-1382509292.cos.ap-beijing.myqcloud.com/obsidian/20260120201826634.png)
LangChain现在将自己定位成一个基于LangGraph的一个快速构建可靠代理的工具，而它的核心组件之一Agent更是设计成了LangGraph教程中搭建的ReAct Agent（看到这我老泪纵横，当时学LangGraph的时候一直不理解为什么一个多智能体框架要去设计单智能体内部的逻辑）。听说LangChain团队也调研了传统工作流方式失败的原因：冗余封装太多，有的时候实现一些功能自己造轮子甚至更简单；更重要的原因是，市面上比较成功的Agent产品几乎都基于ReAct模式，因此索性直接将LangChain改成了现在这样。
不了解ReAct的朋友不用担心，就理解成一个总会先推理，再选择一个合适的方式行动的大模型就好。具体的设计思路可以参见下图。
![image.png](https://bailu-1382509292.cos.ap-beijing.myqcloud.com/obsidian/20260120202349773.png)
可以发现其实看上去也很简单，如果用小学学过的看图说话来说，不就是大模型（model）有了工具（tool）吗？事实也差不多就是这样，和“工作流”相比，有一种大道至简的感觉。
因此这样的结构就很强调Tool的重要性了，Agent的上限就取决于Tool的质量怎么样。巧的是，Tool又是人来写的，这不就是给大模型“戴上镣铐”吗？但是和之前的最大区别在于，大模型拥有了自主选择工具的权利，使得它真正有了Agent的灵魂——具有了自我规划思考和行动的能力。
从“镣铐”的视角来看，其实LangChain的“tool”和Anthropic提出的“claude skills”在理念上是契合的。前者还在无人问津中挣扎，后者俨然已是Agent开发的领军人物了，想来也是唏嘘。真要说LangChain团队提出这样的设计理念比Anthropic更早，只不过当时它被放在LangGraph的一个新手教程里，没人意识到它的通用性。
## 那LangGraph呢？
本来想再结合以下今天白天我的开发经历再谈谈现在的LangGraph的，但写到这发现这篇文章已经有点长了。只好简单说一些，之后找时间一定出一篇比较硬核的文章，配合代码片段，详细地解析LangGraph的构建哲学，以及诸多优秀的特性，让它得以取缔自己的前身，成为LangChain团队的押注。
## 写在最后的话
本来下午发现LangChain这一重大改变的时候，几乎陷入了一种奇怪的兴奋，似乎取得了什么重大的发现，脑中已经开始飘过一些标题党的示例了。在北京寒冷的风走着，我慢慢冷静下来——首先，肯定早有人发现，至少作者发布大更新的时候，是这个世界最早知道这件事的人。==想起小时候的自己，为了展示那点课外积累，总喜欢找老师请教问题，并有意无意地将对话引向我已知的领域，好在老师面前”显摆“一下。==其实也挺好，至少我现在还有小时候那股热忱在。
于是最后我还是选择了这样一篇近乎散文的形式，写下了这篇文字，不为别的，只为了向那个爱思考、爱总结的自己留下一点东西。如果它也恰好能让你重新想一想：你是在替模型规划一切，还是只是在为它准备舞台，那就再好不过了。
