# **实验名称：TLB 容量与访问开销测量实验**
## 一、实验目的

1. 理解虚拟内存机制中TLB（Translation Lookaside Buffer, 地址转换旁路缓冲寄存器）的作用；
2. 掌握通过程序行为间接测量硬件缓存特性的方法；
3. 测量当前机器上 TLB 的容量（以页数为单位）和 TLB 未命中带来的性能开销；
4. 学会动态获取系统页大小，编写可移植的底层性能测试程序。

---

## 二、实验原理

现代操作系统采用**虚拟内存管理**，程序访问的是虚拟地址。每次内存访问需将虚拟地址转换为物理地址，该过程依赖**页表**。为加速地址转换，CPU 提供了 TLB——一种高速缓存，用于存储最近使用的页表项（Page Table Entry, PTE）。

- **TLB 命中**：地址转换在 CPU 内部完成，延迟低；
- **TLB 未命中**：需遍历多级页表（从内存中读取），延迟显著增加。

由于 TLB 容量有限（通常几十到几百项），当程序同时访问的**不同内存页数量超过 TLB 条目数**时，会发生频繁的 TLB 未命中，导致平均内存访问时间急剧上升。

本实验通过**控制访问不同内存页的数量**，测量访问延迟变化，从而推断 TLB 容量。

## 三、实验要求

### 1. 获取系统页大小

- **不得硬编码页大小**（例如 `#define PAGESIZE 4096` 或直接在代码中写死 `4096`）；
- **必须在程序运行时动态获取当前系统的实际页大小**，推荐使用 POSIX 标准函数 `sysconf(_SC_PAGESIZE)`，也可使用 `getpagesize()`（但前者更具可移植性）；
- **在程序开头打印出检测到的页大小**（单位：字节），以便验证实验参数正确性。

> 💡 **为什么不能硬编码？**  
> 虽然 x86-64 Linux 系统通常使用 4KB 页，但 ARM64（如 Apple Silicon, AWS Graviton）、RISC-V 等架构可能默认使用 16KB 或 64KB 页。即使你仅在本机运行实验，**TLB 测量的正确性完全依赖于“每次访问落在不同物理页”这一前提**。若步长小于实际页大小，多次访问可能命中同一页，导致无法触发 TLB 未命中，从而**完全测不出真实的 TLB 容量**。

> ⚠️ **注意**：`sysconf(_SC_PAGESIZE)` 返回的是**标准小页（base page size）**，这正是 `malloc` 分配内存所使用的粒度，适用于本实验。它不是系统调用，而是 C 库函数，在运行时返回编译期已知的常量，**无性能开销**。

>😊 sysconf和你的应用程序以及操作系统内核的层级关系如下：
``` text
你的程序
   │
   ▼
sysconf()        ← POSIX C 库函数（在 glibc、musl 等中实现）
   │
   ▼
[ 可能调用系统调用，也可能直接返回常量 ]
   │
   ▼
Linux 内核（仅在必要时介入）
```
### 2. 分配跨多页的大数组

- 使用 `malloc()` 或 `mmap()` 分配一个足够大的连续内存区域（建议 ≥ 1024 页）；
- 确保数组跨越多个**不同的物理页**（按页对齐访问即可）；
- **在正式测量前，遍历并初始化整个数组**，以触发所有必要的缺页异常（page faults），避免其干扰后续 TLB 性能测量。

> 💡 **为什么必须初始化？**  
> 刚分配的虚拟内存（无论是 `malloc` 还是 `mmap`）**并未立即映射到物理页**。首次读写某一页时，会触发 **缺页异常（page fault）**，由内核分配物理内存。这个过程耗时远高于 TLB 未命中，若在计时阶段发生，会严重污染性能数据。因此，需在**测量开始前**主动触碰每一页（如写入 0），完成“预热”（warm-up）。

> ⚠️ **如何确保访问的是“不同物理页”？**  
> 只要每次访问的地址间隔 ≥ 系统页大小（如 4096 字节），且起始地址按页对齐，那么每次访问必然落在**不同的虚拟页**。在标准小页配置下，每个虚拟页对应一个独立的 TLB 条目，从而能有效探测 TLB 容量。**无需关心物理页是否重复**——TLB 缓存的是虚拟页到物理页的映射，只要虚拟页不同，就会消耗 TLB 条目。

> 🔧 **`malloc` vs `mmap` 如何选？**
> 
> - **`malloc` 更简单**：适合快速原型，但返回地址不一定页对齐（不过只要访问步长 = 页大小，仍能跨不同页）。
> - **`mmap` 更可控**：天然分配页对齐的内存，且可显式指定 `MAP_POPULATE`（Linux 特有）来提前触发缺页，避免手动初始化。
> 
> 对于本实验，两者均可。若使用 `malloc`，建议分配 `(num_pages + 1) * page_size` 并从第一个页对齐地址开始访问，以规避边界问题。

### 3. 设计访问模式

- 编写一个函数，接受两个参数：
    - `num_pages`：要访问的不同页的数量；
    - `iterations`：重复访问的轮数（用于平滑噪声）；
- 访问策略：依次访问第 0 页、第 1 页、…、第 `num_pages - 1` 页的**首地址**（或其他固定偏移），确保每次访问触发一次地址转换；
- 每次访问执行一个简单的读-改-写操作（如 `*ptr += 1`），防止编译器优化掉访问。

### 4. 精确计时

- 使用高精度计时器（如 `gettimeofday()`、`clock_gettime(CLOCK_MONOTONIC, ...)`）；
- 仅对**内层访问循环**计时，排除初始化和函数调用开销；
- 对多次迭代取平均时间，减少抖动影响。

### 5. 控制干扰因素

- 使用 `taskset` 将程序绑定到单个 CPU 核心，避免调度迁移；
- 在正式测量前进行一次“预热”（warm-up）访问，消除首次缺页和缓存冷启动影响；
- 关闭 CPU 频率调节（如设置为 performance 模式）可选，但需在报告中说明环境。

## 四、实验步骤

- 编写 C 程序 `tlb.c`，实现上述功能；
-  编译程序：

```bash
gcc -O2 -o tlb tlb.c
```

- 在终端中运行程序，依次测试不同 `num_pages`（如 2, 4, 8, 16, ..., 512）下的平均访问时间，固定 `iterations = 10000`；
- 将结果保存为表格（CSV 或文本）；
-  使用 Python/Matlab/Excel 绘制 **“访问页数 vs 平均访问时间（微秒）”** 曲线图；
-  分析曲线拐点，确定 TLB 容量；
-  （可选）估算 TLB 未命中的额外开销（未命中时间 - 命中时间）。

## 五、实验输出示例

程序运行示例：

```bash
$ ./tlb 16 10000
Detected page size: 4096 bytes
Average access time for 16 pages: 1.85 microseconds
```

数据表示例：

|Pages|Avg Time (μs)|
|---|---|
|2|0.42|
|4|0.43|
|8|0.45|
|16|0.48|
|32|1.92|
|64|3.75|

图表应显示：在 16 → 32 页之间出现显著性能拐点，表明 TLB 容量约为 **16 项**。

## 七、注意事项

- 切勿使用 `printf` 等 I/O 操作干扰计时；
- 避免访问超出分配内存范围，防止段错误；
- 若使用大页（Huge Pages），需特别说明，但本实验默认使用标准页；
- 实验应在空闲系统上运行，减少后台干扰。
